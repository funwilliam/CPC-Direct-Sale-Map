# COLLABORATION — AI 協作協議

> 上限 60 行｜Claude（PM+主開發）與 Codex（協作開發者）動工前必讀

## 文件分層（下層不得牴觸上層；牴觸 = 停工修上層）

| 層 | 文件 | 行數上限 | 變更規則 |
|---|---|---|---|
| L0 | PRD.md | 150 | 凍結；改動需產品擁有者批准 + ADR |
| L1 | ARCHITECTURE.md / adr/NNN-*.md | 100 / 40 | ADR 一決策一檔、永不改（推翻用新 ADR） |
| L2 | spec/*.md | 120 | 模組動工前凍結；發現漏洞先補 spec 再實作 |
| L3 | tasks/T-NNN.md | 30 | 派工時建立，完成標記 done，不刪 |

超過行數上限 = 膨脹警報，該次 PR 不得合併。

## 分工

- **Claude**：PRD/spec/ADR、架構骨架、MapAdapter 介面、爬蟲解析邏輯、全部 review。
- **Codex**：領任務卡實作——UI 元件、純函式、測試、CSS、依 schema 寫 fetch 腳本。
- **Codex 禁止**：改 spec/介面、新增 npm 套件（需回報 Claude 開 ADR）、無卡動工。

## 任務卡模板（固定格式，禁止自由發揮）

```markdown
# T-NNN 標題
依據: spec/xxx.md §節
範圍: 允許改動的目錄/檔案
做: 具體行為描述
不做: 明確排除項
驗收: [ ] 可驗證條件（測試/手動步驟）
```

交付時 Codex 在卡片底部附：改動檔案清單、測試結果、偏離卡片之處（若有）。

## 驗收（Claude review 只看三件事）

1. 符合 spec 契約。2. 驗收條件全綠。3. 無超範圍改動（有 = 直接退回，不討論代碼品質）。

## 熔斷機制

觸發（任一）：同卡返工 ≥2 次｜spec 有洞需要「猜」｜超範圍改動｜文件超限。
動作：停工 → 判定問題層級（卡？spec？架構？）→ 修正該層文件 → 必要時 revert → 重開卡。
**原則：永遠先修文件再修代碼。** 每 Phase 結束做膨脹審計（檔案數/測試數/文件行數 vs 上限）。
